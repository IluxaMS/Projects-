### Задание 1: «Конвертер Валют с Историей»
Напишите программу, которая имитирует работу простого конвертера валют.
У вас есть базовый курс обмена (например, 1 USD = 90 RUB).

**Требования к логике:**
1.  Создайте функцию, которая принимает сумму в долларах и возвращает сумму в рублях.
2.  Внутри этой функции должна быть возможность применить комиссию (по умолчанию 0%), если она передана вторым аргументом. Если аргумент не передан, комиссия равна 0.
3.  Реализуйте проверку: если переданная сумма отрицательная или не является числом, функция должна вернуть `null` и вывести предупреждение в консоль.
4.  Создайте вторую функцию (стрелочную), которая принимает результат конвертации и выводит его в красивом формате: «Конвертация завершена: [Сумма] RUB».
5.  Продемонстрируйте разницу в поведении при передаче примитивного типа (числа) и объекта (например, объекта настроек `{ rate: 92 }`) в функцию: попробуйте изменить переданный объект внутри функции и проверьте, изменился ли он снаружи.
6.  Используйте условный оператор `??` для обработки случая, когда курс не передан.

*Цель: отработать объявление функций, параметры по умолчанию, возврат значений, стрелочные функции и понимание передачи по значению/ссылке.*

### Задание 2: «Генератор Уникальных ID с Защитой»
Вам нужно создать систему генерации уникальных идентификаторов для пользователей, которая хранит внутреннее состояние (счетчик), недоступное из глобальной области видимости.

**Требования к логике:**
1.  Используйте механизм замыкания: создайте функцию, которая возвращает другую функцию-генератор.
2.  Внутренняя функция при каждом вызове должна увеличивать счетчик на 1 и возвращать строку вида `USER-[номер]`.
3.  Счетчик должен быть инкапсулирован (скрыт) внутри замыкания и не должен иметь прямого доступа извне (нельзя сделать `counter = 100` глобально).
4.  Создайте два независимых генератора (например, для пользователей и для заказов), используя одну и ту же фабричную функцию. Убедитесь, что их счетчики не пересекаются.
5.  Оберните весь код создания генераторов в самовызывающуюся функцию (IIFE), чтобы никакие вспомогательные переменные не засоряли глобальную область видимости.
6.  Внутри IIFE попробуйте объявить переменную через `var` и через `let` в разных блоках и продемонстрируйте разницу в их доступности за пределами этих блоков (через комментарии или логику).

*Цель: глубокое понимание замыканий, IIFE, области видимости (Function Scope vs Block Scope) и инкапсуляции данных.*

### Задание 3: «Рекурсивный Обход Вложенной Структуры»
У вас есть данные в виде вложенного дерева (например, структура папок или категорий товаров), где каждый элемент может содержать массив дочерних элементов.

**Пример структуры:**
```javascript
const data = {
    id: 1,
    name: "Root",
    children: [
        { id: 2, name: "Child 1" },
        { 
            id: 3, 
            name: "Child 2", 
            children: [
                { id: 4, name: "Grandchild" }
            ] 
        }
    ]
};
```

**Требования к логике:**
1.  Напишите рекурсивную функцию, которая обходит эту структуру и собирает все имена узлов в один плоский массив.
2.  Функция должна уметь работать с любой глубиной вложенности.
3.  Обязательно предусмотрите базовый случай выхода из рекурсии (когда у узла нет детей или массив детей пуст).
4.  Добавьте параметр-фильтр: функция должна принимать необязательный аргумент `minLength`. Если он передан, в итоговый массив попадают только те имена, длина которых больше этого значения.
5.  Продемонстрируйте работу Hoisting: объявите вспомогательную функцию-помощник (которая делает рекурсию) ниже основного кода вызова, но убедитесь, что код работает.
6.  Используйте метод массива (например, `forEach` или `map`) внутри рекурсии для прохода по детям.

*Цель: рекурсия, работа со сложными структурами данных, параметры функции, Hoisting, методы массивов.*

### Задание 4: «Система Логирования с Переопределением»
Создайте модуль логирования, который позволяет гибко менять способ вывода сообщений (в консоль, в alert, или в скрытый массив) без изменения основного кода программы.

**Требования к логике:**
1.  Объявите функцию `logger(message)`, которая по умолчанию просто выводит сообщение в консоль с префиксом `[INFO]`.
2.  Реализуйте функцию `setLoggerMode(mode)`, которая принимает строку (`'console'`, `'alert'`, `'storage'`).
3.  В зависимости от режима, функция `setLoggerMode` должна **переопределять** поведение функции `logger`:
    *   `'console'`: стандартный `console.log`.
    *   `'alert'`: вывод через `alert` (или эмуляцию через подтверждение).
    *   `'storage'`: сохранение всех сообщений в специальный массив (историю), доступный для чтения, но не выводимый сразу.
4.  Протестируйте систему: отправьте несколько логов в режиме консоли, затем переключите режим на `'storage'`, отправьте еще несколько, и затем выведите накопленную историю.
5.  Используйте стрелочную функцию для быстрой проверки сообщения перед логированием (например, если сообщение пустое — не логировать).
6.  Обратите внимание на область видимости массива истории: он должен быть доступен функции `logger` после переопределения, но не быть глобальной переменной (используйте замыкание или блочную область).

*Цель: переопределение функций, динамическое изменение поведения, замыкания для хранения состояния, стрелочные функции.*

### Задание 5: «Финальный Босс: Конфигуратор Игрового Баланса»
Разработайте систему настройки игрового баланса, которая использует все изученные концепции для управления множителями опыта, уровнями и событиями.

**Входные данные:**
Объект конфигурации с начальными значениями и массив событий (действий игрока).

**Алгоритм работы:**
1.  **Инициализация (IIFE + Замыкание):**
    Создайте самовызывающуюся функцию, которая возвращает объект с методами: `getMultiplier()`, `setMultiplier(val)`, `calculateXP(baseXP)`.
    Внутри этого замыкания хранится текущий множитель опыта (приватная переменная).
    Метод `setMultiplier` должен валидировать данные: если значение меньше 1, устанавливать 1.
2.  **Рекурсивный расчет уровней:**
    Напишите рекурсивную функцию `getLevel(xp)`, которая определяет уровень игрока.
    Логика: Уровень 1 — это 0-100 XP. Каждый следующий уровень требует на 50 XP больше предыдущего порога. Функция должна возвращать номер уровня.
    *Подсказка:* Используйте рекурсию с накоплением порога.
3.  **Обработка событий (Передача по ссылке/значению):**
    Создайте функцию `processEvent(playerData, event)`, где `playerData` — объект игрока.
    Функция должна изменять опыт игрока в объекте (помня о передаче по ссылке).
    Если событие критическое, функция должна временно переопределить глобальную функцию расчета бонуса (создайте функцию `calcBonus`, которая по умолчанию возвращает 0, но внутри `processEvent` может быть переопределена на возврат 100 очков).
4.  **Генератор отчетов (Hoisting + Стрелочные функции):**
    Реализуйте функцию формирования итогового отчета. Объявите вспомогательную функцию форматирования строки ниже основного кода (проверка Hoisting).
    Используйте цепочку стрелочных функций для применения множителей: `base -> * multiplier -> + bonus`.
5.  **Демонстрация работы:**
    *   Создайте игрока.
    *   Примените серию событий.
    *   Измените множитель через приватный метод.
    *   Покажите, как менялся расчет XP.
    *   Попробуйте обратиться к приватной переменной множителя извне (должно быть невозможно).
    *   Преднамеренно создайте ситуацию с некорректными данными (например, передача строки вместо числа в рекурсию), отловите ошибку или обработайте её, чтобы рекурсия не ушла в бесконечность.

*Цель: комплексное применение IIFE, замыканий (инкапсуляция), рекурсии со сложной логикой, переопределения функций на лету, понимания ссылочных типов и Hoisting в реальном сценарии.*
